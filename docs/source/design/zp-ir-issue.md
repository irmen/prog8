# Zero-Page Variable Initialization in IR-based Code Generation

## The Problem

Zero-page variables (allocated to addresses $02-$FF for fast access) that have
compile-time initial values (strings, arrays) need special handling at program
startup. The old 6502 code generator handles this, but the gap between what
the IR provides and what the code generator needs makes this hard to implement
in an IR-based backend.

## How the Old 6502 Codegen Handles It

The old codegen (`ProgramAndVarsGen.kt`) has an `entrypointInitialization()`
method inside the `main.start` subroutine that:

1. **Determines which ZP variables have init values** by querying the
   `VariableAllocator.zeropageVars` map -- the actual allocation result from
   the ZP allocator that runs during code generation.

2. **Emits runtime copy code** for each such variable:
   - Stores the initial value data at a ``_init_value`` label (in ROM/data section)
   - Generates code to copy from that ROM label into the ZP address at startup
   - Uses `strcpy` for strings, `sys.memcopy` for arrays

3. **Example emitted code:**
   ```asm
   ; zp str and array initializations
   lda  #<varname
   ldy  #>varname
   sta  P8ZP_SCRATCH_W1
   sty  P8ZP_SCRATCH_W1+1
   lda  #<varname_init_value
   ldy  #>varname_init_value
   jsr  prog8_lib.strcpy
   ```

## What the IR Provides

The IR symbol table has `IRStStaticVariable` with these relevant fields:

- `zpwish: ZeropageWish` -- the programmer's intent (REQUIRE, PREFER, DONTCARE,
  NOT_IN_ZEROPAGE), written at compile time
- `initializationValue: IRVariableInitializer?` -- the init value (Str, Array,
  or Numeric)
- `inBss: Boolean` -- whether the variable is in BSS (zero-page vars are NOT
  in BSS)
- `dirty: Boolean` -- whether the variable needs re-initialization

Notably **missing** from the IR:

- **No `isZp` or `zpAddress` field** -- the actual ZP allocation decision is not
  recorded. The IR only knows the programmer's *wish*, not the final allocation.
- **No relationship between ZP status and initialization value** -- the IR
  doesn't express "this ZP variable needs a runtime init copy from ROM".

## Why This Is Hard

### 1. ZP Allocation is a Backend Responsibility

In the old codegen, `VariableAllocator` makes the final ZP allocation decision
at assembly-generation time. It scores variables by usage frequency, type size,
and loop depth, then assigns actual ZP addresses. This allocator runs inside
the 6502 backend, not in the shared IR generation pipeline.

An IR-based code generator would need to either:
- **Reimplement ZP allocation** (duplicating the allocator logic per backend)
- **Have the IR pipeline perform ZP allocation** and store results in the IR
  symbol table (adding `zpAddress` etc.)

### 2. The Init Copy Code is Generated by the Backend

The `jsr prog8_lib.strcpy` / `jsr sys.memcopy` approach is target-specific
and uses ZP scratch registers (P8ZP_SCRATCH_W1). This code is emitted by the
backend inside `main.start`, not by the IR pipeline. The IR `globalInits`
chunk only handles numeric init values (converted to LOAD+STOREM instructions
in `changeGlobalVarInits()`).

### 3. No Precedent in the IR Pipeline

Looking at `IRCodeGen.kt` lines 111-139 (`generateRomableInits`):

```kotlin
is IRVariableInitializer.Str, is IRVariableInitializer.Array -> {
    // Strings and arrays: keep inline data but now marked readonly.
    // They can serve as ROM-based read-only data.
}
```

For romable mode, strings/arrays with init values are left as inline data
(readonly). Non-romable mode doesn't handle them at all -- they stay as
assignment statements in the AST which become regular instructions.
Neither path encodes the ZP-copy-init pattern into the IR.

## Possible Solutions

### Option A: Add ZP allocation info to the IR

Add fields to `IRStStaticVariable`:
```kotlin
val zpAddress: UInt? = null   // set if allocated in zero-page
```

The IR pipeline would need to run ZP allocation (or the IR file would need
to record the allocation from the earlier AST-based allocator). The backend
can then check `zpAddress != null && initializationValue != null` to emit
the copy code.

### Option B: Emit the copy pattern as IR instructions

Have the IR codegen emit LOAD+STOREM or a new `COPY` instruction for each
ZP string/array init, using temporary registers and scratch ZP addresses.
This would make the copy visible in the IR and any backend can translate it.

### Option C: Leave it to each backend

Each backend reimplements ZP allocation and emits the copy pattern.
Simple but duplicates logic across backends.

### Option D: Record the init as a data section entry

Instead of runtime copy, store the initialized ZP variables' values in a
special data section with metadata (target ZP address, size). The backend
emits a startup copy loop for all entries in this section. This is more
generic but less efficient.

## Current Status

The new6502gen code generator does NOT handle ZP string/array init. All
zero-page scratch registers are hardcoded to fixed addresses, and no ZP
allocation scoring is implemented. Any program using `%zeropage` with
initialized strings/arrays in ZP will get incorrect behavior.
